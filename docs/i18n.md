## Architektura i18n

System oparty jest na **custom dictionary** — nie używamy zewnętrznej biblioteki jak i18next. Działanie:

- Słownik to płaski obiekt `klucz → tekst` dla każdego języka.
- Funkcja `useTranslations(lang)` zwraca `t('klucz')`, który szuka tekstu w danym języku, a przy braku — fallback do polskiego (`defaultLang = 'pl'`).
- Routing: `pl` bez prefiksu (`/o-nas`), `en` i `de` z prefiksem (`/en/about`, `/de/about`).
- Middleware przekierowuje `/` → `/pl`.

Wspierane języki: **PL** (domyślny), **EN**, **DE**.

## Jak dodać nowy język

Załóżmy, że dodajesz język **fr** (francuski):

### Krok 1 — Słownik UI

W `src/i18n/ui.ts`:

1. Dodaj nazwę języka w obiekcie `languages`:
   ```ts
   export const languages = {
     pl: 'Polski',
     en: 'English',
     de: 'Deutsch',
     fr: 'Français',
   };
   ```

2. Skopiuj cały blok `en` jako `fr` i przetłumacz wartości:
   ```ts
   fr: {
     'nav.home': 'Accueil',
     'nav.about': 'À propos',
     // ... wszystkie klucze
   }
   ```

> **Ważne:** Wszystkie klucze muszą istnieć we wszystkich językach. Brakujący klucz = fallback do polskiego.

### Krok 2 — Slug translations

W `src/i18n/utils.ts` dodaj mapowanie adresów dla nowego języka:

```ts
fr: {
  '/about': '/o-nas',
  '/services': '/oferta',
  '/downloads': '/pliki',
  '/forms': '/kwestionariusze',
  '/contact': '/kontakt',
  '/pricing': '/cennik',
  '/faq': '/faq',
},
```

I odwrotną mapę w `pl` (jeśli chcesz linkować z PL do FR):
```ts
pl: {
  '/o-nas': '/about',
  // ...
  '/faq': '/faq',
}
```

### Krok 3 — Strony Astro

Każdy plik w `src/pages/[lang]/` używa `getStaticPaths()`. Upewnij się, że zwraca nowy język.

Przykład w `src/pages/[lang]/index.astro`:
```ts
export function getStaticPaths() {
  return Object.keys(languages)
    .filter((lang) => lang !== 'pl')
    .map((lang) => ({ params: { lang } }));
}
```

To automatycznie wygeneruje `/fr/`, `/fr/about/`, `/fr/contact/` itd.

> Strona `src/pages/index.astro` to tylko PL — nie zmieniaj jej.

### Krok 4 — `pageIdMap` na każdej stronie

Każda strona `[lang]/*.astro` ma mapowanie slugów kolekcji `pages`:

```ts
const pageIdMap: Record<string, string> = {
  pl: 'o-nas',
  en: 'about',
  de: 'uber-uns',
  fr: 'a-propos',
};
```

> **Wyjątek:** `src/pages/[lang]/index.astro` nie zawiera klucza `pl` — używa `?? 'home'` jako fallback. Dla nowego języka wystarczy dodać `fr: 'startseite'` (lub odpowiedni slug).

Stwórz odpowiedni plik w `src/content/pages/a-propos.md`:
```md
---
title:
  name: "À propos"
  slug: "a-propos"
lang: fr
description: "Découvrez igawo.pl — notre mission et notre équipe."
---

Contenu de la page...
```

### Krok 5 — Treści w kolekcjach

Dla każdej kolekcji (`services`, `downloads`, `forms`, `faq`, `testimonials`, `partners`, `posts`) stwórz wersje językowe.

**Konwencja nazewnictwa plików:**
- PL (domyślny): `rozliczenie-z-niemiec.md`
- EN: `german-tax-return-en.md` lub `german-tax-return.md` z `lang: en`
- DE: `steuererklaerung-deutschland-de.md` lub z `lang: de`
- FR: `declaration-impots-france-fr.md` lub z `lang: fr`

**Filtrowanie w komponentach** (np. `ServicesBento.astro`):
```astro
const services = allServices
  .filter((s) => s.data.lang === lang)
  .slice(0, 5);
```

> Komponenty używają pola `lang` ze schematu kolekcji, a nie sufiksów nazw plików. Wyjątek: `FormsSection.astro` nadal stripuje sufiks `-en`/`-de` z `id` przy dopasowywaniu ikon i kolorów (`form.id.replace(/-en$|-de$/, '')`).

### Krok 6 — OG images

W `src/pages/[lang]/index.astro` dodaj ścieżkę do obrazka OG:
```ts
const ogImageMap: Record<string, string> = {
  pl: '/images/og-pl.jpg',
  en: '/images/og-en.jpg',
  de: '/images/og-de.jpg',
  fr: '/images/og-fr.jpg',
};
```

> W rzeczywistości `src/pages/[lang]/index.astro` nie definiuje klucza `pl` — fallback to `'/og-image.jpg'`. Dla nowego języka dodaj `fr: '/images/og-fr.jpg'`.

Wygeneruj obrazek `public/images/og-fr.jpg` (1200×630).

## Słownik UI

### Konwencja kluczy

Klucze używają notacji kropkowej — grupa.konkret:

```
nav.home              → nawigacja
hero.title            → sekcja hero
contact.form.email    → formularz kontaktowy, pole email
meta.home.description → meta tag description dla strony głównej
```

### Typowe grupy

| Grupa | Przeznaczenie |
|-------|---------------|
| `nav.*` | Linki w nawigacji i stopce |
| `common.*` | Stringi globalne (read more, back, skip) |
| `hero.*` | Sekcja Hero — tytuły, CTA, badge’e |
| `about.*` | Sekcja O nas — teksty, statystyki |
| `process.*` | Sekcja procesu — 4 kroki |
| `contact.*` | Formularz i dane kontaktowe |
| `downloads.*` | Sekcja plików do pobrania |
| `forms.*` | Sekcja kwestionariuszy |
| `cta.*` | Sekcja wezwania do działania |
| `services.*` | Nagłówek sekcji usług |
| `partners.*` | Sekcja partnerów |
| `testimonials.*` | Sekcja opinii |
| `search.*` | Wyszukiwarka (placeholder, wyniki, błędy) |
| `404.*` | Strona błędu 404 |
| `cookie.*` | Banner i ustawienia cookies |
| `privacy.*` | Polityka prywatności |
| `terms.*` | Regulamin |
| `newsletter.*` | Sekcja newslettera |
| `meta.*` | Meta descriptions |

### Dodawanie nowego klucza

1. Dodaj do **wszystkich** języków w `ui.ts` (nawet jeśli tymczasowo po polsku).
2. Użyj w komponencie:
   ```astro
   const t = useTranslations(lang);
   <h1>{t('new.section.title')}</h1>
   ```
3. TypeScript podpowie błąd, jeśli klucza nie ma — słownik ma `as const`.

## slugTranslations

`slugTranslations` w `src/i18n/utils.ts` mapuje adresy URL między językami. Używane przez `translatePath()` do przełączania języka w nawigacji.

**Struktura:** każdy język ma mapę `moja-ścieżka → ścieżka-w-języku-docelowym`. Dla języków obcych mapa wskazuje na ścieżkę polską (domyślną). Dla polskiego mapa wskazuje na angielską odpowiednik (używane jako pośrednik przy translacji np. PL → DE).

Przykład — przełącznik języka w Header:
```astro
<a href={translatePath(Astro.url.pathname, lang, targetLang)}>
  {languages[targetLang]}
</a>
```

Dla `/en/about` + target `de` → `translatePath` zwróci `/de/about`.
Dla `/de/about` + target `pl` → zwróci `/o-nas`.

## Treści w kolekcjach

Każda kolekcja w `src/content/config.ts` ma pole `lang: z.enum(['pl','en','de'])`. Po dodaniu nowego języka musisz rozszerzyć enum:

```ts
lang: z.enum(['pl', 'en', 'de', 'fr']).default('pl'),
```

### Schemat title

Wszystkie kolekcje używają zagnieżdżonego title:
```md
---
title:
  name: "Rozliczenie z Niemiec"
  slug: "rozliczenie-z-niemiec"
lang: pl
---
```

W komponencie dostęp przez `service.data.title.name` (tekst) lub `service.data.title.slug` (do URL).

## Strony Astro

### Wzorzec strony podstrony

```astro
---
import { getEntry } from 'astro:content';
import Layout from '../../layouts/Layout.astro';
import { languages, useTranslations } from '../../i18n/utils';

export function getStaticPaths() {
  return Object.keys(languages).map((lang) => ({ params: { lang } }));
}

const { lang } = Astro.params;
const t = useTranslations(lang as keyof typeof languages);

const pageIdMap: Record<string, string> = {
  pl: 'o-nas',
  en: 'about',
  de: 'uber-uns',
};
const pageEntry = await getEntry('pages', pageIdMap[lang] ?? 'o-nas');
const pageDesc = pageEntry?.data.description ?? t('meta.about.description');
const { Content } = pageEntry ? await pageEntry.render() : { Content: null };
---

<Layout title={t('nav.about')} description={pageDesc}>
  <h1>{t('nav.about')}</h1>
  {Content && <Content />}
</Layout>
```

### `pageIdMap` — mapowanie kolekcji `pages`

Każda strona w `[lang]/` mapuje swój język na slug w kolekcji `pages`. Nazewnictwo jest dowolne, ale trzymaj się konwencji:
- PL: oryginalny slug (`o-nas`, `pliki`, `kontakt`)
- EN: angielski odpowiednik (`about`, `downloads`, `contact`)
- DE: niemiecki odpowiednik (`uber-uns`, `dateien`, `kontakt`)

## Komponenty sekcji

Każda sekcja przyjmuje `lang: string` i używa `useTranslations(lang)`:

```astro
---
interface Props {
  lang: string;
}
const { lang } = Astro.props;
const t = useTranslations(lang as 'pl' | 'en' | 'de');
---

<section>
  <h2>{t('about.title')} <span>{t('about.titleHighlight')}</span></h2>
</section>
```

> **Ważne:** Typ `lang` w `useTranslations` musi być rozszerzony przy nowym języku:
> ```ts
> const t = useTranslations(lang as 'pl' | 'en' | 'de' | 'fr');
> ```

## Meta tagi

Słownik `ui.ts` zawiera grupę `meta.*` z descriptionami dla każdej strony:

```ts
'meta.home.description': '...',
'meta.about.description': '...',
'meta.services.description': '...',
```

Na stronach `[lang]/*.astro` meta description pobierany jest w kolejności:
1. `pageEntry?.data.description` (z kolekcji `pages`)
2. Fallback: `t('meta.X.description')`

## Sprawdzenie builda

Po każdej zmianie i18n:

```bash
npm run build
```

Sprawdź w logach:
- Czy wszystkie trasy `[lang]` zostały wygenerowane?
- Czy nie ma błędów `Cannot find module` lub brakujących kluczy?
- Czy `dist/` zawiera foldery `/en/`, `/de/` (i nowy język)?

Przykład poprawnego wyjścia:
```
▶ src/pages/[lang]/index.astro
  └─ /en/index.html (+2)
  └─ /de/index.html (+2)
```

## Znane ograniczenia

1. **Hardkodowane stringi poza `ui.ts`** — Wiele komponentów i stron używa ternarnego operatora zamiast słownika. Przy dodawaniu nowego języka trzeba zaktualizować warunki w:
   - `src/pages/[lang]/faq.astro` — CTA na dole strony
   - `src/pages/[lang]/cennik.astro` — nazwy pakietów, opisy, cechy, CTA
   - `src/components/Footer.astro` — tytuły kolumn, linki, copyright
   - `src/components/NewsletterSignup.astro` — stany przycisku
   - `src/components/CookieBanner.tsx` — etykiety zgód
   - `src/pages/[lang]/o-nas.astro`, `kontakt.astro`, `oferta.astro` — `homePath`
   - `src/layouts/Layout.astro` — locale w meta tagach

2. **Mapowanie ikon w `FormsSection.astro`** — Komponenty główne filtrują treści po polu `data.lang`, ale `FormsSection` nadal używa `form.id.replace(/-en$|-de$/, '')` do dopasowania ikon i kolorów. Nowy język wymaga dodania odpowiednich wpisów w `iconMap` i `colorMap`.

3. **Brakujące klucze w EN/DE** — W `ui.ts` klucze `nav.privacy`, `nav.terms`, `cookie.*`, `privacy.*`, `terms.*` oraz `newsletter.*` są obecne tylko w bloku `pl` (część `newsletter.*` w PL jest poza obiektem `pl`, co powoduje, że fallback do polskiego również dla nich nie działa). Przy dodawaniu nowego języka upewnij się, że te klucze są zdefiniowane we wszystkich językach.

4. **Keystatic CMS** — jeśli używasz Keystatic do edycji treści, nowy język wymaga dodania go w konfiguracji CMS (osobny krok poza Astro).

5. **Middleware** — tylko jedna reguła: `/` → `/pl`. Nie ma auto-detect z przeglądarki.
